JAXWS ${IMPL_VERSION}${IMPL_VERSION_SUFFIX}  is rename of JAXWS 2.0.1 M1 to allow some crticial feature such as standard WS-Addressing support in JAXWS, JAXB 2.1 API and type substitution. The development of JAXWS ${IMPL_VERSION} ${IMPL_VERSION_SUFFIX} is being done on the mr-21 branch in JAXWS CVS workspace.

This release brings in the highly pluggable framework thru abstractions such as Pipe and Message. Refer to the latest architecture document. JAXWS ${IMPL_VERSION} ${IMPL_VERSION_SUFFIX} has many bug fixes made for interoperability with .NET 3.0. This is made mostly possible due to WSIT, which uses JAXWS plugabble architecture to plug in support for WS-Metadata, WS-Policy, WS-ReliableMessaging, WS-Transaction, WS-SecureConversation and WS-Security.

The documentation/samples included with JAX-WS that is distributed on Java.net documents how to use JAX-WS in a non-Java EE 5 servlet container using an implementation specific deployment descriptor sun-jaxws.xml and servlet com.sun.xml.ws.transport.http.servlet.WSServlet.

This means that you can run JAX-WS RI applications in any servlet container that has been enabled with the JAX-WS RI. Applications that use the proprietary DD and servlet will run in a JAX-WS RI enabled Java EE 5 servlet container, but they will be non-portable. If you wish to use JAX-WS in a Java EE container in a Java EE portable manner you need to use the standard Java EE 5 deployment descriptor; please refer to the Java EE 5 or Glassfish documentation/samples. The majority of the documentation included with JAX-WS is valid with Java EE 5 as well.

Installation Instructions

The following applies to the JAX-WS standalone bundles.

Once the JAX-WS ${IMPL_VERSION} ${IMPL_VERSION_SUFFIX} bundle is extracted, it needs to be installed on the Application Server.

Make sure AS_HOME is set to the installation directory of the Application Server. You need to invoke ANT_HOME/bin/ant install command from the root directory of the extracted bundle to install JAX-WS ${IMPL_VERSION} ${IMPL_VERSION_SUFFIX} binaries on the Glassfish application server. However this step is required if you intend to build and deploy a Web service on the Application Server.

To install JAXWS ${IMPL_VERSION} ${IMPL_VERSION_SUFFIX} on Tomcat, simply copy lib/*.jar from the root of JAXWS extracted bundle to $TOMCAT_HOME/shared/lib directory.

If you intend to only invoke an already existing Web service, then alternatively you can use the wsimport and wsgen scripts in the bin directory of the JAX-WS ${IMPL_VERSION} ${IMPL_VERSION_SUFFIX} bundle.

Current Limitations

  • The java.util.Collection classes cannot be used with rpc/literal or document/literal BARE style due to a limitation in JAXB. However, they do work in the default document/literal WRAPPED style.

Known Bugs/Issues

  • Defects Submitted at IssueTracker
  • Propagation of W3CEndpointReference from client to the service endpoint doesn't work.
  • swaRef header types dont get marshalled/unmarshalled fails to get attachment
  • Although JAX-WS customizations are portable across implementations, the names of WSDL and schema files generated are not specified. Therefore each vendor may and most likely will generate different names. Therefore, when switching between implementations of JAX-WS, it may be necessary to modify your customization files to reflect different WSDL and schema file names.
  • This holds true for different releases of the JAX-WS RI. The JAX-WS RI may in each release change these file names to resolve bugs. If and when this does occur, a developer would need to change the filenames in their customization files to match the new file names.
